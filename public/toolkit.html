Why this toolkit exists

As a repo grows, the biggest risks aren’t the features — it’s drift: broken links, inconsistent slugs, missing tags, duplicated pages, and last-minute surprises where preview and production don’t match. This toolkit is the safety layer that turns those risks into repeatable checks and safe workflows.

  • Fast: small commands you can run daily.
  • Safe: guardrails for risky operations (release prep, rewrites, restores).
  • Reproducible: the same steps work across projects.
  • Explainable: every script has a “when to use” and “what it changes.”

Use this page as your checklist: create → validate → preview → release → deploy → recover.

What's included

  • PowerShell helpers for local environment hygiene, repo setup, and repeatable actions.
  • CI/CD snippets for GitHub Actions + Vercel workflows (checks, caching, safe deploys).
  • Templates for YAML/config patterns you can reuse across projects.
  • Operational runbooks and sanity-check playbooks for calm shipping.

Tip: keep scripts under /tools or /scripts, and treat them like product code: versioned, documented, and used through named actions.

Node.js is optional

This docs site is a static HTML site. It does not require Node.js, npm, or a package lock to build or deploy.

A small number of helper scripts (for example validate-site.mjs and partial injection checks) are written in Node purely for convenience. They are tooling — not runtime dependencies.

  • If you have Node: you can run validation and guardrail scripts locally.
  • If you don’t: the site still works, builds, and deploys normally.

This separation is intentional: content and layout remain simple and portable, while automation stays optional and removable.

The daily workflow (boring-but-powerful)

Here’s the repeatable loop I use for content, docs, and automation updates — the same logic applies whether you’re writing a post, adding a page, or shipping a release.

  1. Create the change (page/post/script) in a clean, predictable folder path.
  2. Validate structure and metadata (links, slugs, tags, indexes) before you ship.
  3. Preview locally so you catch issues before they hit production.
  4. Release in a batch when it’s ready (changelog, versioning, clean history).
  5. Deploy with the correct environment settings and a known-good baseline.

The goal is not “more automation.” The goal is fewer surprises.

Guardrails (when mistakes are expensive)

These are the safety patterns that keep production stable and recovery fast. If something goes wrong, you should be able to return to a known-good state quickly.

  • Pre-ship checks: catch broken links, collisions, and missing metadata early.
  • Point-in-time restore: maintain a reference build/tag you can always revert to.
  • Safe history operations: avoid unrecoverable mistakes when rebasing or rewriting.
  • Release discipline: ship in clean batches with a changelog and predictable diffs.

Rollback & recovery (when something breaks)

The goal is simple: if production regresses, you should be able to return to a known-good baseline in minutes — without guessing.

  • Prefer revert over rewrite: if a bad change shipped, revert commit(s) instead of rewriting history.
  • Use freeze tags as restore points: keep “known-good” tags like v1.0-footer-fix.
  • Recover from a tag: git checkout <tag> → validate locally → ship a fix branch (don’t force-push main).
  • Don’t panic-edit production: fix in Git, redeploy cleanly, and keep diffs explainable.

Practical rule: every stable layout milestone ends with a tag and a deploy. Tags are your calm button.

Release checklist (boring = reliable)

  1. Local sanity: open Home, README, Docs index, Toolkit and verify layout is centered.
  2. Link sweep: click header + footer links and confirm no 404s.
  3. Minimal diff: avoid mixing refactors with content updates in the same commit.
  4. Tag a freeze point: run Tag-Release.ps1 and push the tag.
  5. Deploy: confirm Vercel is building the expected branch/output.
  6. Post-deploy verify: test in an incognito window; confirm theme toggle + nav work.

If you can’t explain what changed in one sentence, the batch is probably too big.
