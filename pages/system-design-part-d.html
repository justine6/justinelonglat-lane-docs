Everything starts from a small set of Git repos and automation scripts. From there, the platform fans out into: the consulting site, this docs site, the blog, and a GitHub Pages footprint that makes static content easy to mirror, archive, and reuse.

The four sites at a glance

The goal is not ‚Äúfour random websites,‚Äù but one brand, many entry points. Each site has a clear job:

  • Consulting site ‚Äî justinelonglaT-lane.com. The main marketing and portfolio surface, built with Next.js and deployed on Vercel. This is where clients land, book intro calls, and browse projects.
  • Docs site ‚Äî docs.justinelonglaT-lane.com. The operational source of truth for how the ecosystem runs: tooling, CI/CD, reference architecture, and system-design series (Parts A‚ÄìD).
  • Blog site ‚Äî blogs.justinelonglaT-lane.com. Long-form deep dives, case studies, and technical storytelling that point back to both the consulting site and the docs.
  • GitHub Pages presence ‚Äî static mirrors and documentation exported straight from Git. This gives a low-friction way to host documentation where engineers already live, and doubles as a backup surface if a primary domain is under maintenance.

Part D focuses on how those four surfaces stay aligned without turning releases into ‚Äúwebsite whack-a-mole‚Äù.

Repos, hosting & routing

Under the hood, the system is intentionally simple: each major surface has a dedicated repo, but they share conventions, scripts, and guardrails.

Repository layout

  • justinelonglaT-lane ‚Äî Next.js consulting site (projects grid, videos, contact funnel).
  • jutellane-docs ‚Äî this documentation site, written in plain HTML + a light CSS layer, with PowerShell scripts for releases.
  • jutellane-blogs ‚Äî static blog engine with posts stored in a clean posts/YYYY/MM/slug/ structure and small scripts to generate indexes.
  • GitHub Pages export repos ‚Äî where selected docs and blog content are pushed as already-built HTML for GitHub Pages to serve.

Hosting & DNS

DNS is managed centrally (IONOS), but all records follow a repeatable pattern:

  • A / CNAME records point justinelonglaT-lane.com, docs.justinelonglaT-lane.com, and blogs.justinelonglaT-lane.com to Vercel.
  • GitHub Pages sites use the classic username.github.io pattern with CNAME files so they can sit behind friendly sub-domains if needed later.
  • SPF, DKIM, and MX records are kept in the same place so email (intro-call confirmations, contact forms) is routed consistently.

The key design decision: no snowflake DNS entries. Every new surface either follows the existing pattern or doesn‚Äôt ship.

CI/CD orchestration across sites

Each site has its own pipeline, but they share the same mental model: feature branches for work, main for production, and automation for deploys.

Consulting site pipeline

  • Hosted on Vercel with main ‚Üí production deploys.
  • Preview deployments for feature branches so visual changes can be reviewed before going live.
  • Quality checks (TypeScript, lint, basic tests) are run locally and in CI GitHub Actions before merges.

Docs pipeline

  • A simple main-only deploy on Vercel. Docs are plain HTML + CSS, so the pipeline focuses more on structure and consistency than heavy builds.
  • PowerShell helpers (for example Tag-Release.ps1, build-styles.ps1) keep releases repeatable and tagged.
  • Selected docs can be exported to a GitHub Pages repo as pre-built HTML, so the same content can be served from two places without re-implementing logic.

Blog pipeline

  • Content is written in structured HTML/Markdown under posts/, with indexes generated by a small Node script.
  • GitHub Actions builds and publishes to GitHub Pages, while Vercel can also serve the same built assets for faster global delivery.
  • New posts follow a lightweight check: front-matter, metadata, and basic HTML validation before publish.

Why this design works

Even though each pipeline is small, the ecosystem behaves like a single platform: push to the right branch, and the correct surface updates. There‚Äôs no ‚ÄúSSH into a server‚Äù step hiding behind the scenes.

Operational guardrails

The same philosophy behind the Lambda rescue patterns shows up here: codify what works, then reuse it.

  • Branch protection ‚Äî main is protected across repos. Changes flow through pull requests so history stays clean.
  • Quality checks ‚Äî shared scripts enforce formatting and basic linting before code lands on main.
  • Release tagging ‚Äî important milestones (like the consulting rebrand or the launch of System Design Parts A‚ÄìD) are tagged so you can roll back or audit easily.
  • Documentation first ‚Äî any new pattern (for example, a new CI helper script or DNS change) is written up here in the docs before it is used in production.

Day-to-day runbook

In practice, operating the four sites comes down to a small number of repeatable flows:

Shipping a new case study

  1. Draft the long-form version in the blog repo (deep technical story, diagrams, code samples).
  2. Add a concise project card and project page to the consulting site (outcomes, role, stack).
  3. Capture the ‚Äúhow we ran it‚Äù details here in the docs (tooling, pipelines, and patterns).
  4. Tag the release so future readers can see which version of the platform was used.

Updating shared tooling

  1. Change the script or configuration in a single repo (for example, a new PowerShell helper in the docs toolkit).
  2. Propagate the same pattern to other repos via small, focused PRs.
  3. Update this System Design section so the change is documented in one place.

Handling incidents

If a deploy misbehaves, the recovery path is straightforward:

  • Roll back to the previous tag or commit on main.
  • Open an incident note in the docs repo under System Design.
  • Extract any new patterns and promote them into automation or guardrail scripts, not just tribal memory.

Future evolution

The four-site model is intentionally modest: it proves that a single engineer can run a professional-grade ecosystem by combining good design, small tools, and strong habits.

As the practice grows, this Part D chapter becomes the baseline for:

  • Adding new microsites or landing pages without chaos.
  • Onboarding collaborators quickly ‚Äî ‚Äúread Part D, then follow the runbook‚Äù.
  • Extending the same patterns to client platforms, not just the personal ecosystem.

In other words, the platform is not just a portfolio—it is a live demonstration of DevSecOps, system design, and reliability thinking applied to your own work.
