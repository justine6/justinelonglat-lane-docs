Guardrails (when mistakes are expensive)

These are the safety patterns that keep production stable and recovery fast. If something goes wrong, you should be able to return to a known-good state quickly.

  • Pre-ship checks: catch broken links, collisions, and missing metadata early.
  • Point-in-time restore: maintain a reference build/tag you can always revert to.
  • Safe history operations: avoid unrecoverable mistakes when rebasing or rewriting.
  • Release discipline: ship in clean batches with a changelog and predictable diffs.

Rollback & recovery (when something breaks)

The goal is simple: if production regresses, you should be able to return to a known-good baseline in minutes ‚Äî without guessing.

  • Prefer revert over rewrite: if a bad change shipped, revert commit(s) instead of rewriting history.
  • Use freeze tags as restore points: keep ‚Äúknown-good‚Äù tags like v1.0-footer-fix.
  • Recover from a tag: git checkout <tag> ‚Üí validate locally ‚Üí ship a fix branch (don‚Äôt force-push main).
  • Don‚Äôt panic-edit production: fix in Git, redeploy cleanly, and keep diffs explainable.

Practical rule: every stable layout milestone ends with a tag and a deploy. Tags are your calm button.

Release checklist (boring = reliable)

  1. Local sanity: open Home, README, Docs index, Toolkit and verify layout is centered.
  2. Link sweep: click header + footer links and confirm no 404s.
  3. Minimal diff: avoid mixing refactors with content updates in the same commit.
  4. Tag a freeze point: run Tag-Release.ps1 and push the tag.
  5. Deploy: confirm Vercel is building the expected branch/output.
  6. Post-deploy verify: test in an incognito window; confirm theme toggle + nav work.

If you can‚Äôt explain what changed in one sentence, the batch is probably too big.
